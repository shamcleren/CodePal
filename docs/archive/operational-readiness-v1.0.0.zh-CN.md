# 1.0.0 正式运营缺口清单

这份清单不是路线图，也不是 release notes。

它只回答一个问题：

如果 CodePal 要从“已经能发布的工具”走到“可以持续正式运营的产品”，当前还缺什么。

## 一句话结论

核心功能已经基本够用，当前最主要的欠缺不在监控能力本身，而在：

- 更新分发
- 用户信任
- 诊断与支持

## 必须补齐

这些项目如果不补，`1.0.0` 更像“可下载版本”，还不算“可持续正式运营版本”。

### 1. 软件内更新

目标：

- 用户不需要每次去 `Releases` 手动下载新包
- 应用内可以检查更新、下载更新、重启安装

最低要求：

- 仅 `macOS`
- 仅 `stable` 渠道
- 基于 `GitHub Releases`
- 已签名、已 notarize 的产物可以被 updater 正常消费

### 2. 隐私 / 数据说明

CodePal 会读取本地 session、日志、quota、配置。

正式对外必须明确：

- 会读取哪些本地数据
- 不会读取哪些内容
- 是否会上传任何数据
- 日志里是否可能包含敏感信息
- 用户如何手动删除本地配置 / 缓存 / 日志

最低要求：

- 仓库内一份正式隐私说明文档
- README 或设置页能链接到这份说明

### 3. 日志与诊断导出

正式运营时，问题一定会出现。没有诊断出口，后续支持成本会很高。

最低要求：

- 明确日志位置
- 应用内能复制诊断信息
- 用户可以导出或查看关键错误信息

建议包含：

- 当前版本
- 当前平台
- 启用的集成状态
- 最近一次更新检查状态
- 最近一次 quota / dashboard 同步状态

### 4. 反馈入口

最低要求：

- GitHub Issues 可作为正式反馈入口
- 至少提供 bug report 模板
- 至少提供 feature request 模板

建议模板要求用户附带：

- 涉及哪个 agent / IDE
- 期望行为
- 实际行为
- 是否稳定复现
- 截图 / 日志 / 诊断信息

### 5. 支持范围说明

正式运营必须明确边界，不然用户会默认所有情况都算“受支持”。

最低要求：

- 支持哪些 macOS 版本
- 当前是否仅支持 Apple Silicon
- 哪些 agent / IDE 属于正式支持
- 哪些路径只是当前 calibrated scope

## 建议尽快补

这些不是 `1.0.0` 的绝对阻塞，但越早补，后续运营越轻松。

### 1. 更新失败处理

至少要有：

- 下载失败提示
- 安装失败提示
- 用户如何手动回退到 Releases 的说明

### 2. 应用内关于页 / 更新区块

建议在设置页或关于页里提供：

- 当前版本号
- 检查更新
- 下载状态
- 更新完成后重启安装

### 3. FAQ / Troubleshooting

建议单独整理一页，至少覆盖：

- macOS 首次启动拦截
- Cursor / CodeBuddy 登录失效
- settings.yaml 配置错误
- 无法读取 quota / usage
- 更新失败如何处理

### 4. 错误提示分级

建议把错误至少分成：

- 配置缺失
- 配置错误
- 登录失效
- 数据源异常
- 更新失败

这样用户能更快判断是自己改配置，还是需要提 issue。

## 后续再做

这些适合在 `1.0.x` 或 `1.1` 之后继续补，不建议拖住当前正式版。

- beta / stable 更新渠道
- 更复杂的发布策略
- 匿名 telemetry / 崩溃上报
- 独立官网或 landing page
- 更完整的用户支持与运营工具链

## 当前建议优先级

如果只按最值得投入的顺序排，建议是：

1. 软件内更新
2. 隐私 / 数据说明
3. 日志与诊断导出
4. 反馈入口与 issue 模板
5. 支持范围说明

## 完成标准

当下面这些都成立时，可以更有把握地把 CodePal 当成正式运营中的 `1.0.0`：

- 用户可以在应用内完成更新
- 用户能读到清晰的隐私 / 数据边界说明
- 问题发生时，用户能导出诊断并知道去哪里反馈
- 文档明确写清楚支持范围与产品边界
- 发版流程已经从“人工记忆”收敛成固定清单
